# FH6Auto SArB1e 优化版对比说明

本文档用于说明本仓库 `HikigayaHachiman0211/FH6Auto` 相对原作者仓库 `YOUSTHEONE/FH6Auto` 的主要差异、增强点和适用场景。

> 原作者仓库是 FH6Auto 的基础实现。本仓库是在该基础上围绕稳定性、可恢复性、刷 CR 点、诊断记录和模板覆盖做的深度优化版本。

## 对比基线

- 原作者仓库：`YOUSTHEONE/FH6Auto`
- 本仓库：`HikigayaHachiman0211/FH6Auto`
- 对比基线：`upstream/main`，当前 upstream 版本提交为 `2ad87ae`
- 本地优化版主逻辑规模：`main.py` 约 7240 行；upstream `main.py` 约 3395 行
- 主要变化范围：主流程控制、错误恢复、刷 CR 点、模板缓存、诊断文件、UI 设置、打包脚本和大量图像模板

## 稳健性增强

本仓库重点解决自动化流程长时间运行时最常见的失败点：

- **断点恢复**：模块失败后会尝试重新聚焦游戏、退回漫游或菜单，再从当前模块恢复。
- **进程守护**：定时检测 `forzahorizon6.exe` 是否退出，可配合自动重启命令做恢复测试。
- **跑图兜底**：跑图卡住时先倒车，达到上限后尝试重开赛事，避免单次卡死拖垮整个流水线。
- **刷 CR 守护**：刷 CR 点期间识别手柄失联、赛事设置界面、服务器错误和未知状态。
- **点赞弹窗兜底**：在跑图/刷 CR 卡住时识别点赞按钮，自动点赞后继续，避免蓝图、调教、涂装点赞窗口阻断流程。
- **诊断留痕**：失败时保存当前截图、诊断 JSON、运行日志路径和 GitHub Issues 上报指引，便于复盘和修复。

## 功能多样性增强

本仓库不仅保留原有“跑图 -> 买车 -> 抽奖 -> 移除车辆”的核心流水线，还增加了更适合长时间使用的辅助能力：

- **刷 CR 点独立模块**：支持选择车辆、进入在线劲敌赛事、开启自动驾驶、周期结算并记录 CR 收益。
- **CR 识别与效率统计**：记录开始/周期结算 CR，估算每圈收益和每小时收益。
- **超级抽奖计算器**：可根据目标抽奖数、每圈技术点、每抽技能点计算需要跑图次数。
- **多车辆刷图配置**：支持 S2 900 和 S1 790 跑图车辆配置。
- **更细的筛选与车辆识别模板**：覆盖品牌、等级、驱动、稀有度、赛事、错误弹窗等更多 UI 状态。
- **主页安全提醒**：明确提示 CR 不足可能消耗车辆票券，避免误用贵重物品。

## 维护与排错优势

本仓库的排错设计更适合持续迭代：

- 失败现场默认保存到 `diagnostics/年-月-日/`。
- 运行日志固定写入 `bot_log.txt`。
- 诊断 JSON 会包含模块名、失败原因、计数器、窗口区域、配置快照和截图路径。
- 出错后日志会提示用户提交 `bot_log.txt`、诊断 JSON、错误截图和复现步骤到 GitHub Issues。
- 模板缓存会减少重复读取图片带来的启动和识别开销。

## 适用场景

推荐使用本仓库优化版的场景：

- 需要长时间自动刷技术点、买车、抽奖、移除车辆。
- 需要独立刷 CR 点和周期结算。
- 希望出错时能自动恢复，而不是直接中断。
- 希望失败后留下可定位问题的截图和日志。
- 愿意根据自己的分辨率、画质和游戏状态维护模板图。

更适合使用原作者仓库的场景：

- 只需要最基础的流程自动化。
- 希望代码体量更小、改动更少。
- 不需要刷 CR 点、自动恢复、诊断记录和大量模板兜底。
